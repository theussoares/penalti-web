# Diretrizes e Critérios de Aceitação — Mundo da Sorte Web

## Contexto do Projeto

Plataforma de venda de rifas/títulos de capitalização (**PlataformaRifa
v0.2.46**), desenvolvida pela Play Serviços.

**Stack:** Nuxt 3 · Vue 3 (Composition API) · TypeScript · Pinia · Tailwind CSS
· Vitest

---

## Arquitetura em Camadas

Cada domínio vive em `camadas/`:

```
camadas/
  checkout/        ← fluxo de pagamento e confirmação
  compra/          ← processo de compra de títulos
  escolha-produto/ ← seleção de produto/oferta
  my-purchases/    ← histórico de compras
  ui/              ← componentes compartilhados
  core/            ← funcionalidades base e configurações globais
```

**Regras:**

- Avaliar a camada correta antes de criar qualquer arquivo.
- Usar Grep/Glob para localizar código existente antes de propor algo novo.
- Composables reutilizáveis em `/composables/` (raiz); composables específicos
  dentro da camada.
- Stores Pinia em `/store/` com padrão `useStoreDeX` ou `useXStore`.
- Priorize o desenvolvimento de Dumb Components

## Design System e Padrões de Componentes — Obrigatório

> ⛔ **Nenhuma implementação pode ignorar esta seção.**

### Regras absolutas

1. **Antes de criar qualquer elemento de UI**, buscar componentes, padrões e
   estruturas existentes no projeto.
2. **Nunca criar componentes, estilos ou estruturas que já existam** —
   reutilizar sempre.
3. **Nunca alterar** cores, tipografia, espaçamentos ou qualquer token visual
   definido em `~/assets/css/main.css`.
4. **Toda nova tela ou componente deve ser visualmente indistinguível** das
   telas já existentes — mesma densidade, hierarquia, ritmo visual e linguagem
   de UI.
5. **Toda implementação deve ser baseada em análise prévia** de pelo menos uma
   tela/componente existente do mesmo tipo (tabela, formulário, modal, card,
   etc.).
6. Usar exclusivamente classes Tailwind e CSS variables do projeto. Sem CSS
   inline.
7. Ícones: usar o que já está em uso no projeto; evitar adicionar novas libs de
   ícones sem justificativa. (temos nuxt-icon)

### Referências de padrão

| Elemento       | Onde buscar                                          |
| -------------- | ---------------------------------------------------- |
| Componentes UI | `camadas/ui/` e `/components/`                       |
| Stores Pinia   | `/store/` — ver padrão `useStoreDeCompra`            |
| Composables    | `/composables/` — 50+ exemplos disponíveis           |
| Tipos          | `/types/` — organizados por domínio                  |
| Schemas Zod    | `/schemas/`                                          |
| Breakpoints    | `3xs` (360), `2xs` (375), `xs` (475), `tablet` (520) |

### Checklist de conformidade (obrigatório antes de concluir)

- [ ] Busquei componentes equivalentes antes de implementar.
- [ ] Não criei nada que já existe no projeto.
- [ ] Não alterei nenhum token visual.
- [ ] A nova tela/componente é visualmente consistente com o restante.
- [ ] Sem CSS inline — apenas Tailwind e CSS variables.

## Frontend — Vue 3 + Nuxt 3

- Composition API com `<script setup lang="ts">` em todos os componentes.
- TypeScript estrito: sem `any` sem justificativa; guards para `null/undefined`.
- Props e emits com tipagem explícita: `defineProps<{...}>()` e
  `defineEmits<{...}>()`.
- Stores: sempre via `storeToRefs()` para reatividade correta.
- API calls: composables dedicados; usar `useAsyncData`/`useFetch` conforme o
  contexto SSR.
- Remover imports não utilizados.
- **Componente é apenas apresentação**: nenhum `<script setup>` de componente
  pode conter `$fetch`, `useFetch`, `useAsyncData` ou manipulação direta de
  store (ler/mutar `state`, chamar `actions` sem passar por composable). Toda
  chamada de API e toda lógica de negócio vivem em um composable dedicado; o
  componente só consome o retorno.

### Decomposição obrigatória

- **Nenhum arquivo pode ultrapassar ~300 linhas** — extrair seções em
  componentes.
- Componentes da feature: dentro da respectiva camada.
- Lógica reutilizável em composables dedicados.
- Arquivos existentes acima de ~300 linhas devem ser decompostos antes de
  receber novas features.

### Composables — Responsabilidade Única (SRP)

- **Um composable = uma responsabilidade de domínio.** Separar claramente:
  data fetching, state management, business logic e UI state. Um composable
  que faz fetch + transforma + valida + dispara notificação de UI deve ser
  quebrado (ex: `useProductFetch`, `useActiveProducts`, `useProductActions`).
- Composables podem se compor entre si (`useA` usando `useB` internamente).
- Sem efeitos colaterais implícitos — todo side-effect deve ser explícito no
  nome da função ou documentado.
- Funções internas com no máximo ~20 linhas.
- Sem nomes genéricos (`data`, `res`, `temp`, `aux`) — nomes expressivos que
  expressam intenção (`isLoading` > `loading`, `activeProducts` > `data`).
- Sem funções com mais de 3 parâmetros sem usar objeto de configuração.
- Early return para reduzir aninhamento.
- Sem comentários explicando "o quê" o código faz — nome de função/variável já
  deve deixar isso claro.

```ts
// ❌ ANTES — múltiplas responsabilidades no mesmo composable
export function useProduct() {
  const data = ref(null)
  const loading = ref(false)
  async function load() {
    loading.value = true
    const res = await $fetch('/api/product')
    data.value = res.filter((x) => x.active)
    loading.value = false
  }
  return { data, loading, load }
}

// ✅ DEPOIS — responsabilidade única, composáveis entre si
// useProductFetch.ts
export function useProductFetch() {
  const { data, pending } = useFetch('/api/product')
  return { rawProducts: data, isLoading: pending }
}

// useActiveProducts.ts
export function useActiveProducts(rawProducts: Ref<Product[]>) {
  return computed(() => rawProducts.value?.filter((p) => p.active) ?? [])
}
```

### Idioma do código — Inglês obrigatório

- **Funções, variáveis, constantes, tipos, interfaces, props, emits, refs,
  computeds** → sempre em **inglês**.
- **Comentários de código** → inglês.
- **Nomes de arquivos e pastas** → inglês (`kebab-case` para pastas/arquivos,
  `PascalCase` para componentes Vue).
- **Exceção**: textos de UI (labels, placeholders, mensagens ao usuário) → usar
  sistema de i18n.
- Nunca misturar idiomas em símbolos (ex: ~~`showFormulario`~~, ~~`isAberto`~~).

**Convenção de casing por tipo de identificador:**

| Tipo                    | Convenção                    | Exemplo                       |
| ------------------------ | ----------------------------- | ------------------------------ |
| Componente Vue           | `PascalCase`                  | `UserCard.vue`                |
| Composable               | `use` + `camelCase`           | `useProductList`              |
| Store Pinia              | `use` + `PascalCase` + `Store`| `useCheckoutStore`            |
| Variável / função        | `camelCase`                   | `isLoading`, `fetchProducts()`|
| Constante (valor fixo)    | `SCREAMING_SNAKE_CASE`        | `MAX_RETRY_ATTEMPTS`          |
| Tipo / Interface         | `PascalCase`                  | `ProductPayload`              |
| Arquivo / pasta (não-Vue) | `kebab-case`                  | `order-bump-new.ts`           |

## Internacionalização (i18n)

- `@nuxtjs/i18n` com lazy loading. Locales: `pt` (padrão) e `es`.
- Toda string visível ao usuário usa `$t()` ou `useI18n()` — nunca hardcoded nos
  templates.
- Ao adicionar novas strings, sempre criar a chave nos dois locales (`pt` e
  `es`) em sincronia.
- Arquivos de tradução em `/assets/` e `/i18n/`.

## State Management (Pinia)

Padrão obrigatório:

```ts
export const useStoreDeX = defineStore('x', {
  state: () => ({ ... }),
  getters: { ... },
  actions: { ... }
})
```

Conhecer as stores existentes antes de criar novas:

- `useStoreDeCompra` — carrinho e checkout (store principal)
- `useStoreDeUsuario` — dados do usuário
- `usePaymentStore` / `useCheckoutStore` — pagamento
- `useStoreDeModal` / `useUiStore` / `useStoreDeLayout` — UI
- `useTrackingStore` — analytics

## Processo de Implementação

### Antes de implementar

1. Ler `CLAUDE.md` e mapear a camada/módulo afetado.
2. Usar Grep/Glob para localizar componentes, stores e composables relacionados.
3. Identificar impactos em SSR, i18n e stores existentes.
4. **Alterar um arquivo por vez** → revisar → próximo arquivo.

### Após cada arquivo

- **Estrutura**: camada correta, responsabilidade única.
- **Qualidade**: sem duplicações, imports limpos, tipagens completas.
- **Performance**: sem watchers desnecessários, sem N+1 em composables.
- **Segurança**: sem dados sensíveis expostos, CryptoJS usado corretamente.
- **i18n**: paridade pt/es garantida.

## Critérios de Aceitação

Implementação **concluída** apenas quando:

- [ ] Todas as funcionalidades descritas implementadas.
- [ ] Nenhuma regra ou funcionalidade existente quebrada.
- [ ] Sem componentes ou stores duplicadas.
- [ ] Sem queries ou requests redundantes.
- [ ] Sem violações arquiteturais (arquivo na camada errada, lógica no
      template).
- [ ] Nenhum componente com `$fetch`/`useFetch`/`useAsyncData` ou
      manipulação direta de store — lógica extraída para composable.
- [ ] Composables respeitam responsabilidade única (SRP) — sem
      fetch+transformação+estado de UI misturados no mesmo composable.
- [ ] Identificadores em inglês, seguindo a convenção de casing por tipo.
- [ ] Paridade i18n pt/es completa.
- [ ] `vue-tsc --noEmit` sem erros TypeScript.
- [ ] `npm run lint` sem erros.
- [ ] `npm run test:unit` passando.
- [ ] Experiência visual alinhada ao Design System existente.
